# Slack Off API

Fastify + Prisma backend for Slack Off.

## Local Development

1. Copy environment example:

```bash
cp .env.example .env
```

2. Install dependencies and generate Prisma Client:

```bash
npm install
npm run prisma:generate
```

3. Run local dependencies, then start the API:

```bash
docker compose up postgres redis
npm run dev
```

4. Check health:

```bash
curl -i http://127.0.0.1:3000/health
```

The health payload includes `ok`, `service`, `configSource`, `requestId`, and `traceId`.

## Environment Variables

Required for deployed environments:

- `DATABASE_URL`: PostgreSQL connection string.
- `SUPABASE_JWT_SECRET`: JWT secret used to validate Supabase access tokens.

Common runtime variables:

- `HOST`: bind host, defaults to `0.0.0.0`.
- `PORT`: bind port, defaults to `3000`.
- `LOG_LEVEL`: Pino log level, defaults to `info`.
- `REDIS_URL`: optional Redis URL.
- `CONFIG_SOURCE`: `env` or `nacos`.
- `NACOS_SERVER_URL`, `NACOS_NAMESPACE`, `NACOS_GROUP`, `NACOS_DATA_ID`: required when `CONFIG_SOURCE=nacos`.
- `SENTRY_DSN`: optional error reporting DSN.

Secrets must stay in environment variables or a secret manager. Do not put database passwords, JWT secrets, or tokens in Nacos config JSON.

## Runtime Config

`CONFIG_SOURCE=env` uses defaults from `src/config/runtime.ts`.

`CONFIG_SOURCE=nacos` loads runtime policy values from Nacos and deep-merges them with defaults. Use `config/slack-off-api.nacos.example.json` as a starting point.

## Docker

Build and run the API image:

```bash
docker build -t slack-off-api .
docker run --rm -p 3000:3000 --env-file .env slack-off-api
```

Local smoke stack:

```bash
docker compose up --build
curl -i http://127.0.0.1:3000/health
```

## Database Migrations

Local migration authoring:

```bash
npx prisma migrate dev
npm run prisma:generate
```

Staging/production rollout:

```bash
npm run db:migrate:deploy
npm run start:prod
```

Before production migration:

- take a database backup
- review generated SQL and Prisma migration files
- confirm rollback notes for the release
- run migration first in staging

Use `npm run db:seed` only for controlled reference data such as beans, cosmetics, achievements, and activity templates. Do not run destructive resets against production.

## Logging And TraceId

The API emits structured logs through Fastify/Pino. Authorization headers, tokens, and OTP-like fields are redacted.

Every request has:

- `X-Request-Id`: accepted from the client or generated by the API
- `X-Trace-Id`: accepted from the client or generated by the API
- `spanId`: generated per API request

Responses include `X-Request-Id` and `X-Trace-Id` headers. Error envelopes include `requestId` and `traceId`.

Trace lookup flow:

1. Capture `traceId` from mobile logs or an API response.
2. Search API logs for `trace_id`.
3. Search audit events by `traceId`.
4. Search reward ledger metadata for `traceId`.
5. Inspect related leaderboard, bean, achievement, cosmetic, or activity side effects.

## Deployment Checklist

1. Build image or compile with `npm run build`.
2. Provide production env vars.
3. Run `npm run db:migrate:deploy`.
4. Start with `npm run start:prod`.
5. Verify `GET /health`.
6. Run one traceId smoke test across request logs, audit events, and reward ledger metadata.
